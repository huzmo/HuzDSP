## Overview

HuzDSP is a digital multi-effects guitar pedal, which pairs an STM32H743 microcontroller, a CS4270 audio codec, and an ILI9488 TFT display, running on FreeRTOS with an LVGL-based UI for real-time parameter control and effect chaining.

The goal of this project was to build a complete, professional-grade embedded audio platform rather than a single fixed-function pedal, with a 24-bit/48kHz signal path and a modular effects architecture that supports chaining multiple effects together, configuring them through UI, and extending the effect library without touching the core audio engine. ## Hardware

### Audio Signal Chain

Fundamentally, the signal flows through a linear path: beginning as an analog input from an external source, the signal passes through a buffer stage and is filtered with a cutoff at 27.6KHz to eliminate frequencies above the audible band. The filtered signal is then digitized by the ADC stage of the codec, after which the MCU reads and processes the data in real time before it is finally output and filtered back to analog on the way out.

### Inputs / Outputs

- 2x mono instrument inputs
- 2x mono instrument outputs
- 1x navigation encoder
- 3x parameter encoders
- 1x master volume potentiometer
- 480x320 LCD
- USB-C connector, reserved for future audio input/output to enable full audio interface functionality
- DC barrel jack for 9V-12V supply

### Power

The board is powered through a DC barrel jack and accepts 9V–12V, center-negative. Full supply voltage is used for the analog input/output op-amp stages. A reference voltage at half-supply is generated by a buffered resistor divider for biasing the op-amp stages. This rail is then stepped down to +5V for the codec and +3.3V for the remaining circuitry using DC-DC switching regulators, placed on the opposite end of the board from sensitive analog componentry to minimize switching noise coupling.

### PCB / Physical Design

A 4-layer board was chosen for this design, as it allows for larger ground planes with shorter return paths. A dedicated power layer simplifies routing and improves overall performance.

Physically, the board is designed to act as a shield for the LCD display; the mechanical design is simplified by having the board slot into the display and have the mounting holes lined up. This reduces the amount of wires present in the design and minimizes the length of the path the SPI and GPIO signals have to travel to reach the display processor.

To aid signal integrity, analog, power, and digital circuitry are separated from one another, and high-speed digital traces are routed away from analog sections. A solid ground plane sits directly below the first signal layer to reduce coupling between layers.

### Effect Framework

One of the core design features of this project was to incorporate a framework for modular effects; effectively being able to add as many different kinds of effects as long as a simple template is followed. By decoupling parameter and state data from the effect's process function, an arbitrary process function can be passed into the effect loop without needing to manage unique local variables, previous states, or per-effect initialization logic elsewhere in the codebase.

Each effect is split into two parts: a structure that stores an assignable process function pointer, an array of parameters, and a block of state. The process function can be assigned freely at runtime and represents the per-sample algorithm used to achieve the desired effect. A pointer to the effect structure is passed into the process function, which reads the appropriate parameters and writes to the state block to hold constants, previous sample values, or any other data needed on the next sample cycle.

This framework not only allows for an intuitive, modular approach to adding new effects, but also saves memory by allocating each effect slot at compile-time and assigning the process function pointer at runtime, rather than dynamically creating and destroying effect instances during operation. This also guarantees exactly one instance of each effect exists in memory; if multiple effect slots use the same effect type, their function pointers all reference that single shared instance, regardless of how many slots use it.

### Memory Map

Memory is partitioned into two logical regions. Uncached allocations begin at `0x30000000` and are reserved for the audio path, while cacheable memory starting at `0x24000000` is used for display buffers. For the large display buffers, cacheability is a major advantage, as it allows for low-latency renders for frames which have not been flagged for change by LVGL. As audio data is a constant stream of changing values, caching would result in corrupt data, and continuously caching and uncaching would increase latency rather than reduce it. Therefore, audio buffers are allocated in a region which is strictly uncacheable. Each area can be read freely, which keeps managing state between the audio and display stacks simple.

### Drivers / Peripherals

- **LCD**: ILI9488 driver, adapted and optimized from an existing open-source base for this project's needs
- **Codec**: Custom CS4270 driver, written from scratch (I2C register configuration, reset sequencing)
- **SAI**: Used to read and write audio data to/from the codec
- **I2C**: Used to configure codec registers
- **SPI**: Used to drive the ILI9488 LCD display
- **Master volume potentiometer**: Interfaced by polling and filling a DMA buffer with data inside a dedicated task
- **Encoders**: Interfaced using hardware timers in encoder mode; push buttons are handled via interrupts

## Challenges

### Real-Time DSP Constraints

Running multiple DSP effects in a chain on a single sample-by-sample loop means every effect has a fixed time budget before the next audio sample is due. Effects like the flanger and chorus, which rely on buffered history and interpolation, had to be profiled and optimized carefully to stay within that budget without introducing audible glitches or dropouts.

### Noise / Signal Integrity

Keeping the analog signal path clean while running high-speed digital logic and switching power regulators on the same board meant analog, digital, and power circuitry had to be kept separate from one another, with high-speed digital routed away from analog sections. Coil whine from the TPS563200 buck converter was identified as one source of analog noise on the current board revision, and has been noted as a target for layout improvements in a future hardware revision.

### Resource Limitations

Balancing the audio buffers, display framebuffers, and RTOS task stacks within the STM32H743's memory map required explicit partitioning between cacheable and non-cacheable regions, along with careful DMA buffer placement, to avoid cache-coherency issues between the CPU and the DMA-driven audio and display peripherals.

### Modular Effect Integration

Designing the effect framework so new effects could be added without modifying the core audio engine took a few iterations to get right. The function-pointer-based dispatch approach, paired with per-effect parameter and state arrays, ended up being the design that struck the right balance between flexibility and the deterministic, low-overhead execution real-time audio processing demands.
